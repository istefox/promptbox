# Plan: Context variables (FR-10)

Reference: `SPEC.md`, `docs/adr/0005-context-variables.md`. TDD plan; each task lists goal, files, and the test that proves it.

- [x] **Task 1: Pure `isContextVariable` classification in the domain parser**
  - Goal: one pure source of truth for "this placeholder name is reserved," usable both to exclude `@`-prefixed variables from the modal and to select which parsed variables go to workspace resolution. `parsePlaceholders` / `resolvePlaceholders` / `parseSegments` stay otherwise unchanged, and `src/domain/placeholders.ts` stays free of Obsidian imports.
  - Files: `src/domain/placeholders.ts` (add `export function isContextVariable(name: string): boolean`); `tests/placeholders.test.ts` (new `describe("isContextVariable (FR-10.1)")` block).
  - Test (write first, red, then implement, green): `isContextVariable("@selection")`, `("@title")`, `("@date")`, `("@clipboard")` → `true`; `isContextVariable("selection")`, `("")`, `("client")` → `false`; `isContextVariable("@")` → `true` (boundary case: any name starting with `@`, known or not, is reserved per FR-10.1's literal wording, not just the four supported names). Add one regression case proving the parser stays agnostic of `@`: `parsePlaceholders("{{@title|fallback|hint}}")` still returns `{ name: "@title", defaultValue: "fallback", hint: "hint" }` unchanged, documenting that ignoring the default/hint (FR-10.3) is the caller's responsibility, not the parser's. Run `npm test` and confirm every pre-existing case in the file still passes unmodified.

- [x] **Task 2: Workspace and browser resolver module**
  - Goal: isolate every Obsidian- and browser-API read the four reserved names need behind one function, so nothing else ever touches `app.workspace` or `navigator.clipboard` directly for this feature. Implements FR-10.2's four resolution rules and their notice-on-failure behavior, including the deliberate `@selection` vs `@clipboard` empty-value asymmetry: an empty selection is unresolved, a successfully-read empty clipboard is not.
  - Files: `src/ui/context-variables.ts` (new: `resolveContextVariables(app: App, names: string[]): Promise<Record<string, string>>`). Imports `todayISO` from `src/storage/frontmatter.ts` (reused, not modified) and `MarkdownView`, `Notice`, `type App` from `obsidian`.
  - Test: manual smoke, per this project's standing convention that Obsidian-API-coupled `src/ui/*` code carries no vitest coverage. In a dev vault: (a) select text in an open Markdown note, confirm `@selection` resolves to that text with no notice; (b) with no editor focused (library view active), confirm `@selection` resolves to an empty string plus exactly one notice naming `@selection`; (c) with a note open, confirm `@title` resolves to its basename, and with no note open, confirm empty string plus one notice naming `@title`; (d) confirm `@date` always resolves to today as `YYYY-MM-DD`, with no notice, whether or not a file is active; (e) with clipboard content present, confirm `@clipboard` resolves to it; with clipboard access denied or unavailable, confirm empty string plus one notice naming `@clipboard`; separately, with an actually-empty clipboard, confirm it resolves to `""` with **no** notice (the asymmetry case, do not collapse it into the `@selection` rule). Note: `noUncheckedIndexedAccess` (tsconfig) means indexing the internal name-to-resolver map returns `T | undefined`; guard before calling.

- [x] **Task 3: Wire the copy flow**
  - Goal: implement FR-10.2/FR-10.3 end to end inside the single existing entry point, with no change to its exported signature. On a non-empty placeholder parse, split into `contextVars` / `userVars` via `isContextVariable`; resolve `contextVars` by name only, so a context variable's default/hint can never reach the resolver, statically enforcing FR-10.3; skip the modal and copy immediately when `userVars` is empty; otherwise open `VariableModal` with only `userVars` and merge `{ ...contextValues, ...userValues }` into one final `resolvePlaceholders` call on submit. Leave the pre-existing zero-placeholder fast path (FR-4.4) and `copyRaw` (FR-10.4, already correct) untouched.
  - Files: `src/ui/copy.ts` (modify `copyWithVariables` only; update its doc comment to mention FR-10). No changes to `src/ui/variable-modal.ts` (already renders whatever list it is given), `src/ui/library-view.ts`, or `src/ui/quick-picker.ts` (both already call the unchanged exported signature — confirmed by grep, `copyWithVariables`/`copyRaw` each have exactly one call site outside their own definitions before this change).
  - Test: manual smoke against `SPEC.md` §3's acceptance criteria, run through the library view's "Copy with variables" item action once and the quick picker once (FR-10.5, both entry points must be exercised): body `Review {{@selection}} for {{tone|neutral}}` with a selection present → modal shows only "tone", clipboard has the selection and chosen tone substituted; same body with no editor focused → clipboard has an empty string where `@selection` was, one notice fires, "tone" modal still appears; body `{{@date}}` alone → copies instantly, no modal, today's date in the clipboard; `{{@unknown}}` alone → copies instantly, no modal, `{{@unknown}}` verbatim in the clipboard.
