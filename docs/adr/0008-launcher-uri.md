# ADR-0008: `obsidian://promptbox` launcher URI

| | |
|---|---|
| Status | Accepted |
| Date | 3 Jul 2026 |
| Deciders | Stefano Ferri |
| Related | `SPEC.md` (topic `launcher-uri`), ADR-0001, ADR-0002, `docs/competitive-analysis.md` §6 N1, `PROJECT.md` Phase 1.5 |

## Context

Phase 1.5 wants the prompt library reachable from outside Obsidian: OS launchers (Raycast, Alfred, Shortcuts) trigger `obsidian://promptbox?...`, Obsidian foregrounds, the normal copy flow runs, and the result lands on the clipboard (`SPEC.md` FR-13; `docs/competitive-analysis.md` §6 N1: "the library becomes an OS-wide asset without opening the library view first"). This feature is purely additive: no existing command, view, or domain contract changes shape. Two things not already settled by ADR-0001/0002 need deciding here: (1) where the lookup logic lives and how `path`/`title`/`raw` combine, and (2) how the handler behaves in the first seconds after a cold Obsidian launch, when the in-memory index (ADR-0001) may still be empty.

`registerObsidianProtocolHandler(action, handler)` is a `Plugin` method (`node_modules/obsidian/obsidian.d.ts:5028`, same family as `registerEvent`/`registerView` already used in `main.ts`); Obsidian's `Component` unload machinery reclaims it automatically, matching how the codebase already avoids an explicit `onunload` override. The handler receives `ObsidianProtocolData`: `{ action: string; [key: string]: string | 'true' }` — every query parameter arrives pre-decoded as a string, and a bare `raw=true` arrives as the literal string `"true"`.

## Decision

1. **Lookup stays a pure domain function.** `src/domain/launcher.ts` exports `resolveLauncherLookup(prompts: Prompt[], params: { path?: string; title?: string }): LauncherLookupResult`, a discriminated union `{ kind: "picker" } | { kind: "match"; prompt: Prompt } | { kind: "no-match"; source: "path" | "title"; value: string }`. No `obsidian` import — matching every other file under `src/domain/*` (confirmed by inspection: `prompt.ts`, `query.ts`, `transfer.ts`, `slug.ts`, `placeholders.ts`, `draft.ts` are all obsidian-free). `raw` is deliberately not part of its input: it only ever changes *how* `main.ts` acts on the result (copy-raw vs. variable flow, or picker raw-mode), never *which* prompt is chosen — the same separation `src/ui/quick-picker.ts` already applies between item selection and its own raw/variable branching.

2. **`path` wins outright when both `path` and `title` are supplied.** `path` is the index's own key (ADR-0001: one file per prompt, `path` is its identity, and literally the `Map` key inside `PromptIndex`); `title` is consulted only when `path` is absent or blank. Blank/whitespace-only values (e.g. `?path=`) count as not provided. Title matching trims and lowercases both sides before comparing; a tie on `updated` (the tie-break FR-13.2 states) resolves further by `path` ascending for a fully deterministic result — the spec does not name a secondary key, this one was picked because it is cheap, stable, and mirrors the `title-asc` secondary key already used by `src/domain/query.ts`'s sort comparator.

3. **The protocol handler is registered unconditionally at the top of `onload`, but the lookup work waits on a new `indexReady` promise.** A cold OS-level launch of `obsidian://promptbox?...` can start Obsidian itself; Obsidian cannot dispatch to a third-party action like `promptbox` before that plugin's `onload` has registered it (the action is simply unknown to Obsidian until then), so registering at the very top of `onload` — never deferred into `onLayoutReady` — is the earliest, and provably not-later-than-any-dispatch-point, place to register. Separately, `this.index` is only populated once `PromptIndex.scan()` finishes inside the existing `onLayoutReady` block (`main.ts:93-115`), and that can measurably lag a cold start. `main.ts` now tracks `private indexReady!: Promise<void>`, resolved once the first `scan()` settles inside that same `onLayoutReady` callback; the new `handleLauncherUri` method `await`s it before calling `this.index.getAll()`. This is the only behavior change to the existing startup block — the vault-event registrations (`create`/`delete`/`rename`/`changed`) are untouched.

4. **When neither `path` nor `title` is present (both blank), fall back to `PromptQuickPicker`, opened in raw mode when `raw=true` was also given.** Not spelled out by an acceptance criterion, but it keeps `obsidian://promptbox?raw=true` (no path/title) consistent with the existing `copy-prompt-raw` command instead of silently dropping the flag. Flagged in the implementation plan's risk list for a quick human confirm rather than treated as settled fact.

## Alternatives considered

1. **Lookup placement — pure domain function (chosen) vs. inline inside the `main.ts` protocol-handler callback.** Inline is less code short-term, but it can only be exercised by manual smoke testing: there is no test harness for `main.ts`/`PromptboxPlugin` today (confirmed by grep across `tests/`, zero hits), while FR-13.5 explicitly asks for vitest coverage of the lookup, and every other piece of non-trivial matching/decision logic in this codebase (`query.ts`, `transfer.ts`, `slug.ts`, `placeholders.ts`) already lives in `src/domain/*` for exactly this reason. Rejected: it would be the only decision logic in the plugin without a test, breaking a convention this project has kept consistently across five completed tiers.
2. **`path`/`title` precedence — `path` wins (chosen) vs. `title` wins vs. treat "both present" as an error.** `title`-wins was rejected because `path` is the stronger, unambiguous identifier (it *is* the map key `PromptIndex` uses internally); silently preferring the weaker, collision-prone key when a strong one is also present would be a surprising default and harder to justify to a future reader. Treating "both present" as an error (e.g., a dedicated Notice) was rejected as unrequested scope: nothing in FR-13 or its acceptance criteria describes a failure state for this combination, only for "no match" and "neither given" — inventing a third failure mode not in the spec risks breaking a caller who, for readability, always includes both a stored `title` and `path` in a saved launcher shortcut.
3. **Cold-start ordering — register immediately + internal `indexReady` gate (chosen) vs. register immediately with no gate vs. defer the registration itself into `onLayoutReady`.** No gate was rejected outright: `this.index.getAll()` would read the still-empty pre-scan `Map`, producing a spurious "no prompt matching …" Notice (or an empty picker) on exactly the cold-start path this feature exists for — `SPEC.md`'s own purpose statement names cold start as the primary scenario, not an edge case, and a community report corroborates that Obsidian cold starts can visibly lag plugin/vault readiness for URI-triggered actions (see References). Deferring the *registration* itself into `onLayoutReady` was rejected as strictly riskier than the chosen option: Obsidian cannot dispatch a plugin-owned action before that plugin's `onload` has run, so registering at the top of `onload` is provably no later than any plausible dispatch point, whereas registering only after `onLayoutReady` fires adds a real, uncontrolled window — bounded only by however long layout takes — during which a cold-start URI has no handler at all and is most likely simply dropped by Obsidian with no callback and no way for the plugin to react. Gating only the *lookup work* behind a promise keeps registration maximally early while still avoiding the empty-index race; this is corroborated, not just reasoned, by Obsidian's own doc comment on `Workspace.onLayoutReady` ("runs the callback right away if layout is already ready, or pushes it to a queue"), so wrapping that existing call to also resolve a promise cannot regress the current vault-event wiring.

## Consequences

**Positive:** the library becomes reachable from any OS launcher without opening Obsidian first, extending FR-4's "quick reuse" outside the vault boundary per the product intent in `docs/competitive-analysis.md` §6 N1; the lookup is unit-tested and total (every input maps to exactly one of three outcomes, no throw path, matching NFR-8's "never crash" ethos at the domain layer); the cold-start guard is a two-line, general-purpose mechanism (`indexReady`) that any future startup-triggered feature (e.g., a later URI action) can reuse without re-solving the same race; zero new production dependencies and zero network calls, consistent with ADR-0001/0002 and the project's no-network invariant through Tier 7.

**Negative and accepted:** `handleLauncherUri` and the `indexReady` wiring in `main.ts` are UI/plugin glue and stay outside vitest coverage under this project's established testing boundary (only `src/domain/*` is unit-tested; UI glue is manually smoke-tested), so their correctness — especially the cold-start path — rests on a manual pass rather than CI, and that pass is inherently timing-sensitive and harder to reproduce on demand than a unit test. Three behaviors are filled in by this ADR rather than by the spec (path/title precedence when both are given, `raw` honored on the picker fallback, the `path`-ascending tie-break secondary key): reasonable, low-risk defaults, but unverified against product intent until a human confirms them. Mobile delivery of `obsidian://` URIs from OS-level launchers is unverified end-to-end by this feature and continues to ride on Tier 6's existing needs-human mobile-QA item (`PROJECT.md`) rather than being newly closed here — `docs/competitive-analysis.md` §6 N1 itself flags "a mobile URI path exists but the variable modal UX needs care."

**Neutral:** `main.ts` gains one new field (`indexReady`), one new private method (`handleLauncherUri`), and two new imports (`resolveLauncherLookup`; `copyRaw`/`copyWithVariables`); no existing exported method on `PromptboxPlugin` changes signature, and `copyWithVariables`/`copyRaw`/`PromptQuickPicker`'s constructor are consumed at a third call site with the exact shapes their two existing call sites already use (confirmed by grep) — no change to `src/ui/copy.ts`, `src/ui/quick-picker.ts`, or `src/storage/indexer.ts`. The command palette's `copy-prompt`/`copy-prompt-raw` commands are untouched; the URI action is a third, independent entry point onto the same copy flow, not a replacement for either.
