# ADR-0004: Favorite prompts as a tolerant boolean frontmatter field

| | |
|---|---|
| Status | Accepted |
| Date | 3 Jul 2026 |
| Deciders | Stefano Ferri |
| Related | `SPEC.md` (favorites), `docs/competitive-analysis.md` §4 P0, ADR-0001, ADR-0002 |

## Context

`SPEC.md` (FR-9) asks for a favorites feature: a boolean `favorite` frontmatter field, a star toggle on library cards, a star indicator plus favorites-first ranking in the quick picker, a "Favorites" filter chip, and a "favorites first" sort option. ADR-0001 (notes are the single source of truth, frontmatter via the official API only, tolerant parsing per NFR-8) and ADR-0002 (native Obsidian primitives, vanilla TypeScript, one `styles.css`) are binding. The competitive-analysis P0 entry scopes the work to three layers — domain (`Prompt` field, tolerant parsing), storage (write on explicit toggle), UI (library view, filter bar, quick picker) — at effort S; it does not mention the JSON transfer layer (FR-7).

Two spec details drive the design past a routine field addition:

- FR-9.1 requires malformed values to fall back to `false` **without any warning**, unlike every other scalar field in `normalizePrompt`, which pushes a warning on a type mismatch (`invalid tags: expected list or string`, `invalid title: expected string`, etc.). The acceptance criterion is explicit: `favorite: "yes please"` → "treated as false, warning-free render."
- FR-9.3 requires the quick picker to rank favorites above non-favorites **only at equal fuzzy score**, not overall — a tie-break, not a re-ranking.

## Decision

**Domain (`src/domain/prompt.ts`).** Add `favorite: boolean` to `Prompt` as a required field (not `favorite?: boolean`): the field has exactly two meaningful states, so there is no third "unset" state to preserve, unlike `quality`. Add `"favorite"` to `KNOWN_FIELDS` so it never leaks into `custom`. Parse it with a single strict-equality check, `value === true`, and never push a warning — this is the one field in `normalizePrompt` that is intentionally silent on invalid input, per FR-9.1, and the code carries a comment saying so to stop a future contributor from "fixing" it into symmetry with the other fields.

**Storage (`src/storage/prompt-writer.ts`).** A dedicated `setFavorite(app, file, value)` writes through `app.fileManager.processFrontMatter` only, alongside the existing `createPrompt` / `updatePrompt` / `deletePrompt`. Setting `true` writes `favorite: true`; setting `false` deletes the key entirely, following the same "omit empty/default optionals" convention `draftToFrontmatter` already uses for `category`/`tags`/`quality`/`use_case`, so a never-favorited note never gains a `favorite: false` line. `favorite` is outside `PromptDraft` and outside `updatePrompt`'s `OPTIONAL_FIELDS` deletion list, so editing metadata through the modal preserves whatever favorite state is already on disk with no code change in that path.

**Reactivity.** No new event wiring. `processFrontMatter` triggers Obsidian's `metadataCache "changed"` event, already subscribed in `main.ts`, which flows through `PromptIndex.handleCreateOrModify` → `onChange` → each view's existing single render path (ADR-0002). The "reflects within 1 s" requirement (FR-9.2) is inherited from already-shipped, already-verified plumbing (the same path `updatePrompt` uses since Tier 3).

**Filtering and sorting (`src/domain/query.ts`).** `LibraryQuery` gains two independent booleans: `favoritesOnly` (filter, AND-combined with the existing predicates in `runQuery`, included in `isQueryActive`) and `favoritesFirst` (sort modifier). `favoritesFirst` wraps whichever `SortKey` comparator is active rather than multiplying the `SortKey` union — this is what "within each group the current sort order applies" (FR-9.5) means structurally: favorite/non-favorite are two groups, the selected sort still orders within each. Both fields default to `false` in `emptyQuery()`, so "Clear filters" resets them for free.

**Quick-picker ranking (`src/domain/query.ts` + `src/ui/quick-picker.ts`).** A new pure, generic function, `rankFavoritesFirst<T>(items, scoreOf, isFavorite)`, implements the FR-9.3 tie-break: a decorate-sort-undecorate pass that preserves the caller's relative order whenever two items' scores differ, and only reorders (favorite first) when scores are exactly equal. It makes no assumption about which direction of `score` is "better," so it cannot invert Obsidian's own fuzzy ranking. `PromptQuickPicker.getSuggestions` calls `super.getSuggestions(query)` (Obsidian's fuzzy scorer, unmodified) and pipes the result through `rankFavoritesFirst`. `renderSuggestion` calls `super.renderSuggestion(match, el)` first (preserving the built-in match-highlighting via `renderResults`) and then prepends a star marker when `match.item.favorite`.

**Library view (`src/ui/library-view.ts`).** The star toggle is a button placed in the card header, before the title — not grouped with the transient action buttons (copy / edit / open / delete) — because it is a persistent state indicator the user needs to scan at a glance, not a one-off action. It reuses the existing `promptbox-item__action clickable-icon` classes (so it inherits color, hover, and the 44 px mobile touch-target rule for free) plus a new `is-favorite` modifier for the filled look, sets `aria-pressed`, and uses `setTooltip`/`aria-label` exactly like the other item actions. The click handler resolves the `TFile` at click time (matching the existing delete/open-as-note handlers), calls `setFavorite`, shows a `Notice` only on failure, and never calls `render()` directly — the index's existing change subscription is what redraws the view.

**Filter bar (`src/ui/filter-bar.ts`).** One fixed "★ Favorites" chip, reusing the existing `promptbox-chip` / `is-active` pattern, bound to `favoritesOnly`. Unlike the dynamic type/category/tag chip groups, it always renders (never hidden at zero matches): it is a fixed filter dimension, not a value discovered from the index. A "Favorites first" checkbox sits next to the sort dropdown, bound to `favoritesFirst`.

**JSON transfer is explicitly out of scope.** `buildExport`, `validateImport`, and `applyEntry` in `src/domain/transfer.ts` / `src/storage/transfer-io.ts` all construct their `ExportedPrompt` objects from an explicit field allowlist (no spreading of `Prompt`), so `favorite` is dropped on export without any code change. This matches the competitive-analysis scoping (domain/storage/UI only) and keeps `schema_version: 1` untouched. See Alternatives and Consequences for the trade-off.

## Alternatives considered

1. **Store `favorite` inside the existing `custom: Record<string, unknown>` passthrough instead of a first-class `Prompt` field.** Zero change to the `Prompt` interface or `KNOWN_FIELDS`, reusing the mechanism already built for phase-2 namespaced fields. Rejected: `custom` carries no type safety or tolerant-parsing guarantee, so every read site (library view, filter, sort, quick picker) would re-implement its own ad hoc truthiness coercion, risking exactly the kind of cross-site inconsistency (e.g., quick picker treating `"true"` as truthy while the filter bar does not) that `normalizePrompt` exists to prevent. A first-class field costs one `KNOWN_FIELDS` entry and one parse function and centralizes the tolerant-boolean contract exactly once, matching how `quality`, `visibility`, and `tags` are already handled.
2. **Bake "favorites first" into the `SortKey` union (e.g. `favorite-updated-desc`, `favorite-title-asc`, ...) instead of an orthogonal `favoritesFirst` boolean.** Rejected: doubles the `SortKey` enum and the `comparator` switch for a property (favorite-first grouping) that is logically independent of the chosen order; it also does not read naturally as "within each group the current sort order applies" (FR-9.5), which is a composition, not a cross product baked into a single enum value. The boolean-plus-wrapper keeps `SortKey` and its comparator untouched and composes with all four existing values automatically, including any added later.
3. **Reuse the existing `PromptDraft` / `updatePrompt` pipeline for the toggle (add `favorite` to `PromptDraft`, flip it in the edit modal, save).** Rejected: FR-9.2 asks for a single click on the card, and routing a one-bit change through "open modal → find checkbox → click Save" fails the glanceable, sub-1-second interaction the requirement describes, and drags the entire `Setting`-row modal machinery in for a toggle. A dedicated `setFavorite(app, file, value)`, parallel to `deletePrompt`, is the minimal write that matches the existing "one explicit user action, one write function" pattern.
4. **Add `favorite` to the `schema_version: 1` JSON transfer contract (`ExportedPrompt`, `validateImport`, `applyEntry`) for full export/import round-trip fidelity.** Rejected for this iteration: FR-7 already has its own frozen, separately-tested contract and round-trip guarantee (FR-7.4); extending it forces a compatibility ruling this feature does not need to make (does a schema_version bump, or a silently-added optional field, apply to files exported by earlier plugin versions that lack it?) and pushes the effort past the S sizing the competitive-analysis assigned. Accepted trade-off, tracked below as a revisit trigger.

## Consequences

**Positive.** The tolerant-boolean parse lives in exactly one place (`normalizePrompt`), matching NFR-8's contract and the explicit "warning-free" acceptance criterion precisely. Toggling reuses 100% of the already-shipped, already-verified reactive pipeline (`processFrontMatter` → `metadataCache "changed"` → `PromptIndex` → view `render()`), so FR-9.2's "reflects within 1 s" requires zero new plumbing and zero new render paths, preserving ADR-0002's single-render-path discipline. Because `favorite` sits outside both `PromptDraft` and `updatePrompt`'s `OPTIONAL_FIELDS`, `createPrompt`, `updatePrompt`, and `draftToFrontmatter` need no code change at all — the field survives edit-modal saves purely as an emergent property of the existing merge design, which is itself already exercised by `draft.test.ts`'s exact-shape assertions on `draftToFrontmatter`'s output. Deleting the key instead of writing `favorite: false` keeps unfavorited notes exactly as clean as today, preserving the portability acceptance criterion (US-8: a plain, readable note) with no special-casing. The quick-picker tie-break is a pure, generic, fully unit-testable function with zero Obsidian coupling, consistent with the codebase's existing domain/glue split (`runQuery` is the same shape of extraction). Every new CSS need reuses an established class or state-modifier convention (`is-active`, `promptbox-item__action`'s mobile sizing), so the visual surface added is small and low-regression.

**Negative and accepted.** `favorite` does not round-trip through JSON export/import; a library exported and re-imported elsewhere silently loses its favorite marks (Alternative 4). This is accepted for effort-S scope and flagged as a revisit trigger: if a future change touches `schema_version` for another reason (e.g., the N7 curated-packs proposal already sketched in the competitive analysis), `favorite` should be folded in at the same time rather than left as a permanent gap. The silent (no-warning) parse for `favorite` is a deliberate asymmetry against every other field in `normalizePrompt`; without the code comment and this ADR as the record of intent, a future contributor could plausibly "fix" it by symmetry and regress the acceptance criterion. `favoritesFirst` and the picker tie-break each add one extra comparison per sort/compare call; this does not change the asymptotic cost of `runQuery` or the picker's ranking (still O(n log n)), so it carries no realistic NFR-1 risk at the 1,000-prompt target, but it is worth remembering if a future performance investigation looks at either hot path. No command or default hotkey is added for toggling the active note's favorite state; NFR-7 forbids shipping default hotkeys regardless, and SPEC.md does not request a command, so none is built — this can be added later without any architecture change if usage shows demand.

**Neutral.** `LibraryQuery` grows by two fields; since this type is transient view state, constructed fresh per view instance and never persisted to `data.json` or serialized anywhere, the change carries no migration or backward-compatibility concern — verified by grep, its only consumers are `query.ts`, `filter-bar.ts`, `library-view.ts`, and the test files already covered by this plan. Placing the star in the card header rather than inside the item-actions row is a visual-hierarchy choice, not a structural one; it can be revisited in CSS alone without touching the write path or the domain model if a future design pass wants it elsewhere. Toggling favorite status from inside the quick picker itself (FR-9.3, explicitly COULD) is not built now; the picker only displays and ranks by the flag, which keeps this iteration's picker changes read-only and low-risk.
