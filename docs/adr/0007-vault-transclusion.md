# ADR-0007: Wikilink transclusion at copy time via a pure single-pass body assembler

| | |
|---|---|
| Status | Accepted |
| Date | 3 Jul 2026 |
| Deciders | Stefano Ferri |
| Related | `SPEC.md` (vault-transclusion), `docs/competitive-analysis.md` §4 P2b, ADR-0001, ADR-0002, ADR-0005 (sibling branch, cited for a directly analogous pattern only) |

## Context

`SPEC.md` (FR-12) asks the FR-4 copy flow to resolve `[[target]]`, `[[target|alias]]`, and `![[target]]` references inside a prompt body to the linked note's content, at copy time, with a preview step. Depth is capped at 1: inserted content is never re-scanned for further links (FR-12.2), which also means no cycle can ever recurse, so no runtime visited-set/cycle-detection bookkeeping is needed, non-recursion by construction already rules it out. Unresolvable links (missing note, or a `#heading`/`^block` sub-reference, out of scope for partial-content resolution per §5) stay verbatim and are named in one Notice (FR-12.4). A preview modal lists each resolvable link, its target, and the inserted size when at least one link resolves; a body with zero wikilinks copies exactly as today, with no new UI at all (FR-12.5). Copy raw bypasses resolution entirely (FR-12.7). The hardest requirement is FR-12.6: links resolve first, then placeholder substitution runs on the *original* body's placeholders only, "placeholders inside transcluded content are not collected and not substituted... resolved/inserted text is never re-parsed."

That last clause is a correctness requirement, not a nicety, and it cuts both ways in this specific codebase. Promptbox's own prompt notes are full of `{{...}}` syntax, so linking prompt-to-prompt (a realistic, expected usage of this exact feature) means a transcluded note's body can plausibly contain literal `{{name}}` text that happens to match one of the *host* prompt's own placeholder names. A naive two-pass implementation, either "resolve links into an expanded string, then run the existing `resolvePlaceholders` regex over the whole expanded string" or the reverse order, re-scans text it should never touch again in at least one direction: transcluded content re-scanned for `{{...}}` in the first ordering, or a user-typed placeholder *value* re-scanned for `[[...]]` in the second (a value like `[[my-notes]]` typed into a variable field should not itself trigger a fresh transclusion pass). ADR-0005 (context variables, sibling branch, not present in this tree) rejected an analogous "pre-substitution then re-parse" design for exactly this class of bug, there a resolved `@selection` value's own text getting mis-parsed as a new placeholder, and settled on a single-parse-then-merge pipeline. That precedent is a direct, reusable analogy this decision follows deliberately, not an accident of two people reaching the same idea independently.

The existing copy flow (`src/ui/copy.ts`) already receives `app` at `copyWithVariables(app, title, body)`, but not the calling prompt's own vault path. Obsidian's official resolution API, `MetadataCache.getFirstLinkpathDest(linkpath, sourcePath)`, takes a required `sourcePath` used to pick "the best match for a linkpath" (its own doc comment), the only case this affects is a vault with two notes sharing a basename in different folders, but when it applies, an empty or wrong `sourcePath` silently resolves to the *wrong* note's content with no error at all. A prior agent note on this project records a preference to keep exported UI entry-point signatures unchanged so call sites need zero edits; that preference is honored everywhere it costs nothing here (`copyRaw` needs zero changes at all) but is knowingly overridden in one narrow, justified place, see Decision below.

Binding constraints carried over unchanged: ADR-0001 (notes are the single source of truth; this feature writes no frontmatter and adds no persisted state), ADR-0002 (native Obsidian primitives, vanilla TypeScript, a `Modal` for the preview, no framework), no network calls, TypeScript `strict: true` plus `noUncheckedIndexedAccess`, and the project's standing testing-boundary convention: `src/domain/**` is pure and vitest-covered, `src/ui/**` is Obsidian-API-coupled glue verified only by manual smoke test. That boundary is not merely a style preference on this repo, `tests/mocks/obsidian.ts` stubs only `Notice`, `TFile`, and `Plugin`; there is no faked `App`, `MetadataCache`, or `Vault`, so any code that touches `app.metadataCache` or `app.vault` is structurally impossible to unit test today, not just discouraged from it.

## Decision

Split the feature across one new pure domain module, one additive export on an existing pure module, and one new UI module, wired into the existing copy orchestration:

1. **`src/domain/transclusion.ts`** (new, no Obsidian import, vitest-covered). Exports an interface `Wikilink { raw, target, alias, isEmbed, hasSubReference }` and two pure functions:
   - `detectWikilinks(body: string): Wikilink[]`, a regex scan mirroring `placeholders.ts`'s own documented style (innermost match only, no nested brackets, malformed/empty-target constructs skipped and left for verbatim pass-through). `!?\[\[target(#heading|^block)?(\|alias)?\]\]`, char classes excluding `[`, `]`, `|`, `#`, `^` from the target segment so nesting behaves exactly like `{{a{{b}}}}` already does for placeholders, only the innermost well-formed construct matches. A `#`/`^` sub-reference sets `hasSubReference: true` unconditionally; such links are unresolvable by design (§5, out of scope) regardless of whether the base note exists, and this function never attempts to look that up, resolution is entirely the UI layer's job. Returns one entry per occurrence (not deduplicated), because an accurate total-size warning (FR-12.5) needs the true count of insertions, a body linking the same note three times inserts its content three times; callers dedupe by `target` themselves wherever they need a per-note view (the resolution map, the preview's grouped rows). This is a deliberate divergence from `parsePlaceholders`'s dedupe-by-name choice, justified by the two functions serving different downstream consumers (one field per unique placeholder name vs. true total inserted bytes).
   - `assembleBody(body: string, linkContents: Map<string, string>, variableValues: Record<string, string>): string`, the mechanism that satisfies FR-12.6 for real. It computes wikilink match spans (via its own internal scan of `body`, exactly as `resolvePlaceholders` already independently re-scans `body` today rather than reusing `parsePlaceholders`'s matches, an existing, already-accepted pattern in this file) and placeholder match spans (via a new additive export from `placeholders.ts`, point 2 below) over the *same, single, untouched* `body` string, merges both span lists by start offset, and performs one linear slice-and-join pass. A wikilink span is replaced by `linkContents.get(target)` when present (absent, whether missing-note or `hasSubReference`, leaves it verbatim); a placeholder span is replaced by `variableValues[name]` when the key exists (unknown/malformed stays verbatim, identical semantics to today's `resolvePlaceholders`). Because every span is computed from offsets into the one pristine original string, and the assembled output is produced in a single pass with no second `.replace()` over anything already substituted, neither substituted kind can ever be re-scanned by the other, in either direction, this is stronger than the literal "links first, then placeholders" phrasing in FR-12.6, and deliberately so (see Alternatives). Where two matches overlap (a `[[link]]` embedded inside a placeholder's pipe-segment hint, or a `{{placeholder}}` embedded inside a link's alias text, both syntactically legal since `{}` and `[]` are disjoint character classes), the span that starts first wins and consumes the nested one; whichever bracket pair opens first is authoritative, and this is symmetric in both nesting directions, not a special case for one of them.

2. **`src/domain/placeholders.ts`** gains one additive export, a `PlaceholderMatch { raw, start, end, variable: PromptVariable | null }` shape and a `matchPlaceholders(body): PlaceholderMatch[]` function giving positions alongside the existing parse result. `parsePlaceholders` and `resolvePlaceholders` keep their exact existing signatures, return shapes, and behavior; `tests/placeholders.test.ts`'s current cases are not touched. This is the only change to an existing domain file, purely additive visibility, so the risk of a regression here is essentially the risk of a copy-paste error in a new, separately-tested function, not a risk to proven behavior.

3. **`src/ui/transclusion-modal.ts`** (new, Obsidian-coupled, manual-smoke-only). `resolveWikilinks(app, sourcePath, links): Promise<{ resolved: Map<string, string>; unresolved: string[] }>` calls `app.metadataCache.getFirstLinkpathDest(target, sourcePath)` once per unique non-`hasSubReference` target; a hit reads `stripFrontmatter(await app.vault.cachedRead(file))` (the exact existing helper from `src/storage/frontmatter.ts`, reused unmodified, so transcluded content is stripped identically to how a prompt's own body already is, FR-12.3); a miss, and every `hasSubReference` target, is collected into `unresolved`. `TransclusionPreviewModal` is a `Modal` (mirroring `ImportModal`'s shape: a `Setting`-based Confirm/Cancel button pair, a list built with `contentEl.createEl`) listing each unique target with its occurrence count and per-occurrence size, a computed total, and a warning line above 50,000 characters (informational only, per FR-12.5, never a hard block); its confirm button reads "Continue" rather than "Copy", since confirming here does not always immediately copy, a variable-filling modal may still follow. Cancelling closes the modal and calls nothing further, an entire aborted copy, identical in spirit to the existing "Cancel copies nothing" contract on `VariableModal`.

4. **`src/ui/copy.ts`**: `copyWithVariables` gains one new required parameter, `sourcePath: string`, becoming `(app, title, body, sourcePath): void`. The exported return type stays `void` (not `Promise<void>`): the function's entire body becomes `void run(...)` where `run` is a new internal `async` function, exactly the fire-and-forget idiom `main.ts` already uses for `exportPrompts`. This means no call site needs a new `void` prefix, only the one new argument. `run` detects links, resolves them, fires the "unresolved link(s)" Notice *after* a successful clipboard write (not eagerly at detection time), so a cancelled preview reports nothing, matching "cancel copies nothing" extended naturally to "and notifies nothing, since nothing happened". When `resolved.size > 0` it opens `TransclusionPreviewModal` first; its confirm callback (or, when there is nothing to preview, the flow directly) proceeds through the *unchanged* `parsePlaceholders(body)` call and either an immediate `assembleBody(body, resolved, {})` write or a `VariableModal` exactly as today, now writing `assembleBody(body, resolved, values)` instead of `resolvePlaceholders(body, values)`. `copyRaw(title, body)` is untouched, zero lines changed, it already satisfies FR-12.7 by construction (it never parses the body at all).

5. Two call sites need exactly one added argument each, `prompt.path`, nothing else: `src/ui/library-view.ts`'s "Copy with variables" item action, and `src/ui/quick-picker.ts`'s non-raw branch. Grep confirms these are the only two call sites of `copyWithVariables` in the tree (`copyRaw` also has exactly two, both unaffected). This is the one deliberate, narrow exception to the "signature-stable, zero call-site edits" preference recorded from prior work: the alternative (an empty or best-guess `sourcePath`) trades a real, silent correctness bug (wrong-note resolution in any vault with duplicate basenames) for a cosmetic-only convenience, and a one-argument, mechanical, two-file edit is a small enough price that the preference's underlying intent, avoid risky or repetitive call-site churn, is still honored in substance.

6. `styles.css` gains a small `.promptbox-transclusion` rule set (row list, size text, a warning line reusing the existing `--text-warning` variable already used by `.promptbox-item__warning`), and `docs/adr/README.md` gains the ADR-0007 index row, following the exact convention the two prior Phase 1.5 ADRs (0005, 0006, sibling branches) already established.

## Alternatives considered

1. **Two sequential regex passes** (`substituteWikilinks(body, resolved)` producing an expanded string, then `resolvePlaceholders(expanded, values)`, or the same in the opposite order). Rejected: this is what FR-12.6's literal prose describes, and it is the simplest possible code, but it structurally violates the requirement's own stated invariant ("resolved/inserted text is never re-parsed") in a realistic scenario specific to this exact product: a transcluded note that is itself another prompt containing `{{...}}` syntax gets re-scanned by the second pass, and if its literal text happens to contain a `{{name}}` matching one of the host prompt's own collected variable names, it is incorrectly substituted. The reverse ordering has the mirror problem: a placeholder value typed by the user that happens to look like `[[target]]` gets picked up by a wikilink pass running after it. Both directions are narrow but real, and the second one especially plausible (users routinely paste snippets containing bracket-like text into a fill-in form). The chosen single-pass design computes both span sets from the one untouched original string and assembles once, closing off both directions structurally rather than by convention.

2. **Sentinel-token two-phase splice**: replace resolved wikilinks with unique placeholder tokens first, run the existing `resolvePlaceholders` unmodified over the tokenized string, then do a final literal (non-regex) token-to-content replace. Rejected: it is also fully correct, but it adds an extra string-transformation phase and a new hazard class of its own, choosing a token format guaranteed never to collide with real user content, and does not reduce the amount of new code meaningfully versus the chosen design. The single-pass span merge achieves the identical guarantee (inserted/substituted text never re-scanned) with one function and one clear input/output contract, which is also the easier thing to exhaustively unit test.

3. **Detect wikilinks via `MetadataCache.getFileCache(file)?.links`/`.embeds`** (Obsidian's own parsed link cache) instead of a custom regex. Rejected: it requires an indexed `TFile` with an up-to-date cache, an impure, asynchronous dependency that cannot satisfy `SPEC.md` §4's "pure domain function, vitest-covered" constraint for detection; its cached position offsets are relative to the *raw* file content (frontmatter included), not the frontmatter-stripped body string `copyWithVariables` actually receives from `index.getBody`, which would need its own offset-translation step; and `parsePlaceholders` already established this project's own precedent of pure-regex parsing for prompt-body syntax, a second, cache-based mechanism for a structurally similar problem (finding bracketed references in body text) would be an unnecessary second pattern in the same codebase.

4. **Omit `sourcePath`, resolve links with an empty string or the plugin's configured prompts folder as a stand-in.** Rejected: it preserves the "zero call-site edits" preference completely, but `getFirstLinkpathDest`'s own contract (a required, non-optional second parameter, "get the *best match*") signals that argument is load-bearing for disambiguation; using a placeholder value silently mis-resolves a link in any vault where two notes share a basename in different folders, precisely the "never manual path guessing" failure mode FR-12.1 is written to rule out. A one-argument, two-file, mechanical edit is a materially smaller cost than a silent, hard-to-diagnose wrong-note bug.

5. **Resolution and preview logic in a new `src/storage/transclusion.ts`** (mirroring `transfer-io.ts`'s existing role: vault-facing helpers used by the UI layer). Considered seriously since it matches an existing organizational precedent (storage/ talks to the vault, ui/ renders and orchestrates), but rejected in favor of placing it under `src/ui/` because `SPEC.md` §4 states this explicitly and unambiguously ("vault resolution and preview UI live in the UI layer"), and because the testing-boundary consequence is identical either way (this code gets zero vitest coverage regardless of folder, `tests/mocks/obsidian.ts` fakes neither `MetadataCache` nor `Vault`), so there is no testability argument to weigh against the spec's explicit placement.

6. **Depth-limited recursive resolution with a visited-path set**, resolving links inside transcluded content up to some small depth and guarding cycles explicitly. Out of scope per `SPEC.md` §5 and rejected here on cost/benefit grounds beyond that: depth cap 1 (never re-scan inserted content for further links, the same rule that already protects the placeholder invariant) makes cycles structurally impossible without any runtime bookkeeping at all, a visited-set would be solving a problem the depth cap already eliminates by construction.

## Consequences

**Positive.** FR-12.6's strongest, easiest-to-get-subtly-wrong requirement, no re-parsing of resolved or inserted text, is enforced by construction in one small, exhaustively-testable pure function (`assembleBody`) rather than by a convention someone has to remember at every future call site; this mirrors and reinforces the identical engineering judgment already recorded in ADR-0005 for an analogous re-parsing hazard, so the codebase now has a named, repeatable pattern ("single-pass span assembly over the pristine original string") for any future feature that layers more than one substitution over prompt body text. `copyRaw` needs zero changes and stays a trivially-correct escape hatch. A body with zero wikilinks is unaffected in every observable way: `assembleBody` with an empty `linkContents` map degrades to exactly `resolvePlaceholders`'s existing behavior, and the "zero new UI" requirement (FR-12.5) falls out of a single `resolved.size > 0` check in the orchestration function, not a parallel code path that could drift from the tested one over time. Only two production call sites change, each by one added argument, confirmed by grep before writing this decision, so the blast radius of the one signature change is small, mechanical, and fully enumerated up front.

**Negative and accepted.** The single-pass assembler is more code than either naive two-pass alternative, roughly one focused function instead of reusing two existing ones as-is, a deliberate trade of a small amount of extra, well-contained complexity for closing a correctness gap the spec explicitly names. `copyWithVariables` picks up a required fourth parameter, a narrow, acknowledged exception to a previously-recorded "zero call-site edits" preference; the exception is scoped to exactly this one function and justified by a concrete correctness need (link disambiguation in vaults with duplicate note basenames), not a precedent for loosening that preference generally. The entire new UI-layer surface (`transclusion-modal.ts`, plus the changed half of `copy.ts`) gets zero automated test coverage, consistent with, and not a new departure from, this project's existing testing-boundary convention and its underlying tooling limitation (no faked `App`/`MetadataCache`/`Vault`), but it is the single largest untested surface this feature adds, and correctness there rests entirely on the manual dev-vault smoke test enumerated in the implementation plan. Whether heading/block sub-references (`hasSubReference: true`) are named in the same "unresolved link(s)" Notice as genuinely missing notes is an explicit assumption this decision makes, not a literal requirement: `SPEC.md`'s out-of-scope section uses the word "unresolvable" for both cases, which this decision reads as intentional grouping, but a human reviewer could reasonably want them silent instead, this is flagged for Gate 2 review, not silently decided. The 50,000-character warning is informational only; nothing in `SPEC.md` asks for a hard cap, so a user can still copy an arbitrarily large clipboard payload from a large transcluded note, an accepted, spec-consistent trade-off, not a defect.

**Neutral.** `detectWikilinks` returns one entry per occurrence rather than deduplicating by target, the opposite dedup choice from `parsePlaceholders`; both are deliberate and independently correct for what each function's caller needs (unique fields to fill vs. true total inserted volume), not an inconsistency to reconcile. The overlap tie-break rule (whichever bracket construct opens first consumes anything nested inside it, symmetric in both directions) is a real, if deep, edge case with no natural "correct" answer in the spec; this decision picks the simplest self-consistent rule rather than special-casing either direction, and documents it so it reads as intentional rather than discovered later as a surprise. `src/domain/placeholders.ts`'s new additive export (`matchPlaceholders`) does not change how the `@`-prefixed context-variables namespace (sibling branch, ADR-0005, not in this tree) would eventually integrate, that branch's own changes to `copy.ts`'s orchestration are independent of this one, but whoever reconciles the two branches later should check that `copyWithVariables`'s call signature (now four parameters here, likely reshaped independently there) merges cleanly; this is a forward-looking integration note, not a currently-broken interaction.
