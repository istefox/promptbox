# ADR-0012: Related prompts — pure weighted-overlap scorer, surfaced read-only in the edit modal

| | |
|---|---|
| Status | Accepted |
| Date | 3 Jul 2026 |
| Deciders | Stefano Ferri |

## Context

As Promptbox's library grows toward the ~1,000-prompt target (NFR-1), curating the taxonomy (US-5: keeping type/category/tags clean over time) gets harder by browsing alone. `docs/competitive-analysis.md` §6 N6 proposes surfacing each prompt's nearest neighbors by shared tags, category, and title/body token overlap as a differentiator no competitor offers, built entirely on data the plugin already indexes: no new fields, no network, no embeddings.

`PROJECT.md`'s Phase 1.5 row for this feature still carries that original one-line phrasing ("title/body token overlap"). The feature's own `SPEC.md` (approved 2026-07-03, the binding document for this chain per the task brief) narrows scope deliberately: token overlap is computed over `title` and `use_case` only, and body-content similarity is explicitly out of scope ("bodies not loaded in the modal", `SPEC.md` §5). This ADR follows `SPEC.md` as authoritative for the exact scoring formula and surface; `PROJECT.md`'s shorter phrasing is a pre-refinement summary that gets reconciled when the tracker is next updated by the orchestrator, which is outside this ADR's write scope.

Constraints inherited from the accepted architecture: ADR-0001 (notes are the single source of truth; the in-memory index is disposable) means the scorer must run over already-indexed `Prompt` records and never touch the vault directly. ADR-0002 (vanilla TypeScript, native Obsidian primitives) means the surface is a plain DOM block inside the existing `Modal`, not a new component system. The project's established testing boundary — only `src/domain/*`-shaped, Obsidian-import-free code gets vitest coverage; UI glue is manual smoke (confirmed by `tests/` holding no test file for the existing `PromptModal`) — applies unchanged. `SPEC.md`'s FR-19.3 and its constraints section forbid any new frontmatter, settings, or network calls; this is a pure read-side, in-memory feature layered on Tiers 1-3 (domain model, storage, library view), which are already met.

Existing code reviewed for precedent: `src/domain/prompt.ts` (the `Prompt` shape and its `category === ""` "no category" convention), `src/domain/slug.ts` (the diacritics-stripping normalization technique), `src/domain/query.ts` (pure-function-over-`Prompt[]` shape, `localeCompare` tie-break chaining), `src/ui/prompt-modal.ts` (the `PromptModalDeps` narrow-deps pattern, the single-render-path `display()` method, the existing "open note" button), `src/storage/indexer.ts` and `src/main.ts` (how `Prompt[]` snapshots and tag pools already flow from the index into modal deps), and `src/ui/library-view.ts` (the `addItemAction` icon-button pattern and its "stale index" Notice wording).

## Decision

Add a new, Obsidian-import-free domain module `src/domain/related.ts` with two exported pure functions:

- **`similarityScore(a: Prompt, b: Prompt): number`** — FR-18.1's weighted sum, symmetric (`similarityScore(a, b) === similarityScore(b, a)`):
  - **+3 per shared tag.** Tags are compared as deduplicated sets, exact string equality (case-sensitive), matching every other tag-comparison site in the codebase today (`query.ts`'s `tags.every((t) => p.tags.includes(t))`, the modal's chips input). Deduplicating first guards against a tag repeated inside one note's own frontmatter (user-editable, NFR-8) inflating the count.
  - **+2 when `category` matches**, only when both `a.category` and `b.category` are non-empty and equal. `SPEC.md` does not state the empty-category case explicitly; treating two uncategorized prompts as "sharing a category" would be noise (most uncategorized prompts would then falsely relate to each other), and `category === ""` already means "no category" everywhere else in the codebase (`isCustomValue`, the FR-2.6 pill rendering, `PromptDraft`). This ADR makes that reading explicit and binding for the scorer.
  - **+1 per shared token**, where a prompt's token set is built from `title` and `use_case` concatenated, then normalized: Unicode NFD, diacritics stripped, lower-cased, split on runs of non-alphanumeric characters, empty tokens dropped, duplicates collapsed into a set. This mirrors the normalization *technique* `slug.ts`'s `slugify` already uses, but is a small, purpose-built local function that returns a token set rather than one hyphenated string (see Alternative 3 for why this is not a shared import).
  - Weights live as three named constants (`TAG_WEIGHT = 3`, `CATEGORY_WEIGHT = 2`, `TOKEN_WEIGHT = 1`) so the formula in code is traceable back to FR-18.1 by inspection.

- **`relatedPrompts(target: Prompt, all: Prompt[], limit = 5): Prompt[]`** — FR-18.2's ranking: scores `target` against every other prompt in `all` (excluding `target` by `path` equality, the domain's natural key), drops zero-score results, sorts by score descending, ties broken by `updated` descending then `path` ascending (the same `localeCompare`-chaining style as `query.ts`'s comparator), and returns the top `limit` **prompts** — not score-and-prompt pairs (see Alternative 4). `limit` defaults to 5 per FR-18.2.

Surface (FR-19): `PromptModalDeps` (`src/ui/prompt-modal.ts`) gains one new field, `allPrompts: Prompt[]`, a plain snapshot array built the same way `tagPool` already is today — `main.ts`'s `modalDeps()` adds `allPrompts: this.index.getAll()` next to the existing `tagPool: this.buildTagPool()`. `PromptModal`'s constructor computes:

```
this.related = mode.kind === "edit" ? relatedPrompts(mode.prompt, deps.allPrompts, 5) : [];
```

once, stored as a `private readonly related: Prompt[]` field. `display()` — the modal's single render path (ADR-0002) — only ever *reads* `this.related`; it never recomputes it, which satisfies FR-19.2 even though `display()` itself can run more than once per open (the existing "add new taxonomy value" inline flow re-renders the whole form; the related list is untouched by that code path since it's computed in the constructor, not in `display()`).

`display()` renders a "Related" block, guarded by a single `this.related.length > 0` check — which covers both "create mode" (`related` is always `[]`) and "zero matches" (edit mode, nothing scored above zero) absence cases from FR-19.1 with one condition — positioned after the Version/Body fields and before the final Save/Cancel `Setting` row. For each related prompt: its title, a `.promptbox-pill.promptbox-pill--type` badge (existing class, reused unmodified) for `type`, and one icon-only "open as note" action button in the same `setIcon` + `aria-label` + `setTooltip` shape `library-view.ts`'s `addItemAction` already uses. No score is rendered anywhere (FR-19.1's "score-free display").

The action's click handler resolves the `TFile` via `this.app.vault.getFileByPath(relatedPrompt.path)` (the modal already has `this.app`; no new deps method is needed for this), calls `deps.openFile?.(file)` **before** `this.close()` — the same order this file's existing "Open note to edit body" button already uses, kept identical for consistency — and, on a stale/missing file, shows the same Notice wording `library-view.ts`'s `openAsNote` already uses ("Note not found — the index may be stale.") without closing the modal. `styles.css` gets a small, BEM-named addition (`.promptbox-related`, `.promptbox-related__heading`, `.promptbox-related__item`, `.promptbox-related__title`, `.promptbox-related__action`) following the file's existing naming and the `.is-mobile` 44px touch-target convention already applied to `.promptbox-item__action`.

## Alternatives considered

1. **Maintain an inverted tag/category index inside `PromptIndex` for faster-than-O(n) lookups.** This would let `relatedPrompts` avoid a full scan of `all` on every modal open. Rejected: `SPEC.md`'s own constraints section calls "O(n) per modal open at 1,000 prompts... acceptable (NFR-1 scale)"; a full scan at n=1,000 does a handful of set and string operations per candidate — well under a millisecond in total — so there is nothing to optimize yet. Building and incrementally maintaining an inverted index would couple `PromptIndex` (ADR-0001's deliberately disposable, rebuildable cache) to a new similarity concern, growing both its responsibility and its test surface, for a benefit no measurement asks for. This directly parallels ADR-0001's own accepted stance of deferring index-shape changes until NFR-1 measurements demand them.

2. **Share one generic scorer utility with the sibling `feat/tag-category-suggestions` branch's local heuristic scorer (`PROJECT.md` Phase 1.5).** This would avoid maintaining two similar weighted-overlap functions. Rejected: that branch is unmerged and not present in this working tree (`git branch -a` confirms `feat/tag-category-suggestions` exists only as a sibling branch, and the session is explicitly scoped to `feat/related-prompts`). Depending on code that does not exist here, or pre-emptively inventing a shared abstraction whose real shape is still moving on another branch, either breaks the session-scope boundary or risks a shape mismatch that has to be re-reconciled at merge time regardless. A standalone, roughly 40-line scorer here costs little duplication now and can be folded into a shared primitive later, once both shapes are stable and visible in the same tree. This keeps the same *philosophy* the prior agent's note asked for (pure, weighted, deterministic, over normalized tokens) without a premature shared dependency.

3. **Reuse `slugify` from `src/domain/slug.ts` directly for tokenization**, instead of writing a small local tokenizer. This would reuse existing, tested diacritics-stripping code instead of a second copy. Rejected: `slugify`'s contract is "produce one URL/filename-safe hyphenated string, with an `'untitled'` fallback for empty input" (FR-3.1); it is not shaped to return a *set of tokens*, and reusing it would mean splitting its hyphenated output back into tokens (fragile), while also dragging the irrelevant `"untitled"` fallback into a scoring context where an empty title should contribute zero tokens, not the literal token `"untitled"`. A small local tokenizer in `related.ts` that reuses the *technique* (NFD normalize, strip diacritics, lower-case, split on non-alphanumeric runs) but is shaped for token sets is both more correct and still only a few lines.

4. **Return `{ prompt: Prompt; score: number }[]` from `relatedPrompts`** instead of `Prompt[]`, so the modal or a future caller could display or debug the numeric score. Rejected: FR-19.1 explicitly mandates "score-free display", and the project's domain-lookup convention is that pure domain functions return raw data, never anything shaped for display — exposing the score is display-shaped, not data-shaped, and nothing downstream needs it (the modal only needs prompts in ranked order). Keeping the public return type as a plain, ranked `Prompt[]` makes "never show the score" a property of the type itself, not just of current UI code's discipline.

5. **Also surface related prompts as a panel in the library view** (per-card, in addition to the edit modal), so the feature is discoverable without opening the edit modal. Rejected: explicitly out of scope per `SPEC.md` §5 ("a library-view 'related' panel"). It would also be architecturally the wrong place to spend an O(n) scorer: the library view re-renders its entire visible list on every filter change and every search keystroke (`runQuery` inside `library-view.ts`'s single `render()` path), so a per-row related-list would multiply the O(n) scorer by the number of visible rows on every keystroke — directly fighting NFR-1's <100 ms search budget and FR-19.2's "no keystroke recomputation" intent. The edit modal, opened one prompt at a time and held open long enough to read, is the only place in the current UI where a single O(n) computation is naturally amortized over the interaction.

## Consequences

**Positive.** The scorer and ranking function are pure, deterministic, and fully vitest-covered in `src/domain/related.ts` with zero Obsidian imports, matching ADR-0001/0002's domain/UI split and the project's existing testing boundary. No new frontmatter, settings, persisted state, or network calls are introduced, preserving FR-19.3, ADR-0001's "notes are the single source of truth", and the project's zero-network posture. The surface threads exactly one new, narrow field (`allPrompts: Prompt[]`) through the existing `PromptModalDeps` shape rather than the whole plugin or the live index, preserving the modal's narrow-deps testability boundary and its single existing construction site in `main.ts`. A full O(n) scan is cheap enough at the stated 1,000-prompt scale that no premature indexing work is needed, consistent with ADR-0001's own deferred-optimization stance. The UI reuses existing CSS primitives (`.promptbox-pill--type`, the `setIcon`/`setTooltip` action-button pattern, the mobile 44px touch-target convention) instead of inventing a new visual language, so the `styles.css` addition stays small and consistent with the file's established BEM naming. Finally, `relatedPrompts`'s return type (`Prompt[]`) makes FR-19.1's "score-free display" requirement true at the type level: there is no numeric field for the UI to accidentally render.

**Negative and accepted.** There is no stopword filtering on title/use_case tokens, so short or common words can occasionally inflate token overlap between prompts that are not meaningfully related; this is accepted because `SPEC.md`'s acceptance criteria do not test for it, and a stopword list is itself a design decision (which language? user-editable? English-only per NG-6?) better deferred to a real complaint than invented speculatively here. Tag matching is case-sensitive exact-string equality, diverging from the diacritics/case-insensitive rule FR-18.1 states only for title/use_case tokens, so two tags differing only by case ("Code" vs. "code") are never treated as shared; this is accepted because it matches every existing tag-comparison site in the codebase today, and changing tag-matching semantics generally is a larger, cross-cutting change outside this feature's scope. Category matching requires both prompts' `category` to be non-empty and equal, so two uncategorized prompts never earn the category weight; this is an interpretive gap-fill on FR-18.1 (the spec does not state the empty case), made explicit here so a future reader does not mistake it for an oversight, and it is the only reading consistent with how `category === ""` is already treated everywhere else in the codebase. The related list is a frozen snapshot for the lifetime of the open modal (per FR-19.2); if another prompt is edited, created, or deleted through some other open leaf while this modal stays open, the Related section can go stale until the modal is reopened — accepted because it is the direct, generalized consequence of FR-19.2's "no keystroke recomputation" intent, and the modal is normally a short, single-focus interaction. There is no automated coverage for the modal-rendering glue itself, only for the domain scorer; this matches the established project convention, but it means the "open as note closes the modal without corruption" acceptance criterion is verified by manual smoke only, not by a regression test.

**Neutral.** `docs/adr/README.md`'s index table shows a numbering gap between ADR-0003 and ADR-0012 until the eight sibling Phase 1.5 branches (ADR numbers 0004-0011) land; this is an expected artifact of the parallel-branch workflow already in place for this project, not a defect of this ADR, and is called out directly in the updated README so it reads as intentional. The scorer's shape (a weighted pure function over two records, using normalized token sets) intentionally mirrors the philosophy of the sibling `feat/tag-category-suggestions` scorer without sharing code with it; reconciling the two into one shared primitive, if still desired once both branches are visible together, is future work for whoever integrates them, not a concern of this ADR.
