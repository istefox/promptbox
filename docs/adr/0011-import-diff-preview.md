# ADR-0011: Import-conflict diff preview

| | |
|---|---|
| Status | Accepted |
| Date | 3 Jul 2026 |
| Deciders | Stefano Ferri |
| Related | `SPEC.md` (import-diff-preview, FR-17), `docs/competitive-analysis.md` §6 N5, ADR-0001, ADR-0002, Tier 5 (FR-7) |

## Context

Tier 5 (`docs/project.md`) shipped JSON import with a whole-import conflict policy (skip / overwrite / duplicate) and an end-of-run summary (FR-7.3). The overwrite policy is the destructive branch: it replaces an existing note's frontmatter and body with no per-item confirmation. `docs/competitive-analysis.md` §6 N5 flags this as a real gap ("no competitor has import at all," let alone a preview of what it is about to overwrite) and notes it doubles as groundwork for Phase 2's community update flow (CFR-5.2, updating a locally-modified installed prompt).

`SPEC.md` (FR-17) asks for a preview step, gated strictly to the overwrite policy with at least one conflict, that fires before any write and shows, per conflicting path: which metadata fields change (old → new) and a body change indicator (unchanged, or changed with +N/−N line counts from a minimal line-based comparison, not a full patch render). Confirm proceeds with the whole import; cancel aborts with nothing written. Skip and duplicate stay untouched, the end-of-run summary stays unchanged, and `schema_version: 1` is not touched.

Existing building blocks (read in full before this design): `src/domain/transfer.ts` (`ExportedPrompt`, `buildExport`, `validateImport`, `planImport` → `ImportAction[]`), `src/storage/transfer-io.ts` (`runImport`, which computes the set of existing relative paths and then executes `planImport`'s actions), `src/ui/import-modal.ts` (`ImportModal`, which validates and calls `runImport` directly today), and `src/ui/confirm-modal.ts` / `src/ui/variable-modal.ts` (the two existing `Modal` + `Setting` idioms this codebase already uses for confirmation and list-of-fields dialogs).

Two settled project patterns constrain this design and are not re-opened here: only `src/domain/*` gets vitest coverage, UI and storage-layer I/O glue is verified by manual smoke (confirmed by inspecting `tests/mocks/obsidian.ts`, which stubs only `Notice`/`TFile`/`Plugin` — no `App`, `vault`, or `fileManager`, so `transfer-io.ts` and any UI code are not vitest-testable in this harness without a substantial new mock investment that is out of scope for an S-effort feature); and pure domain functions return raw structured data, never pre-formatted user-facing strings, with all presentation left to `src/ui/*`.

## Decision

Add a pure domain diff engine to `src/domain/transfer.ts`, an I/O orchestration function to `src/storage/transfer-io.ts`, and a new preview `Modal` to `src/ui/`, wired into `ImportModal.submit()` as a gate that only engages for the overwrite policy when at least one conflict exists.

**Domain (`src/domain/transfer.ts`, vitest-covered):**

- Extract the per-item mapping already inside `buildExport` into an exported `toExportedPrompt(prompt: Prompt, body: string, relativePath: string): ExportedPrompt`. `buildExport` calls it unchanged; its output is byte-for-byte identical to today (guarded by the existing `buildExport` tests). This gives the diff engine one canonical "Prompt + body → transfer shape" mapping to reuse for the *existing* side of a conflict, instead of a second, drifting field list.
- Add `lineDelta(oldBody: string, newBody: string): { added: number; removed: number }`: a line-multiset difference (`Map<string, number>` of per-line counts, old lines increment, new lines decrement, then sum the positive residues as `removed` and the negative residues as `added`). This is deliberately not an LCS/patch algorithm (FR-17.3 explicitly excludes a full patch render): a pure append of N lines yields `+N/-0` exactly as the acceptance example requires, a single edited line yields `+1/-1`, and byte-identical bodies yield `0/0`. The one accepted blind spot is a pure reorder of otherwise-identical lines, which this metric reports as unchanged; documented as a known limit of a line-based (not order-aware) comparison, matching the spec's own framing.
- Add the discriminated union `ImportFieldChange` (`{ field: "tags"; from: string[]; to: string[] }`, `{ field: "quality"; from: number | undefined; to: number | undefined }`, or `{ field: <the seven remaining scalar keys>; from: string; to: string }`) and `ImportDiff` (`{ targetPath: string; identical: boolean; fieldChanges: ImportFieldChange[]; body: { changed: boolean; added: number; removed: number } }`). Values stay raw (arrays stay arrays, absent quality stays `undefined`); no comma-joining, no "none" fallback labels, no formatted sentences — that is UI work.
- Add `diffImportEntry(existing: ExportedPrompt, incoming: ExportedPrompt): ImportDiff`. Compares exactly the ten fields `applyEntry` in `transfer-io.ts` is able to overwrite — `title, type, category, tags, quality, use_case, visibility, version, created, updated` — in that same order, so the preview is a faithful, complete mirror of what the write is about to touch (including `created`/`updated`: overwrite silently carries the imported entry's dates verbatim today, per the existing `applyEntry` comment, and silently rewinding a note's `updated` date is exactly the kind of surprise this feature exists to surface). `body` comes from `lineDelta` plus a `changed` flag derived from `added > 0 || removed > 0` (not from a separate strict-equality check, so the two never disagree). `identical = fieldChanges.length === 0 && !body.changed`.

**Storage (`src/storage/transfer-io.ts`, manual smoke, no new vitest — see Context):**

- Extract `runImport`'s inline "existing relative paths in the folder" computation into `listExistingRelativePaths(app: App, folder: string): Set<string>`, called by both `runImport` (unchanged behavior) and the new preview builder below.
- Add `buildOverwritePreview(app: App, folder: string, doc: ExportDoc): Promise<ImportDiff[]>`: computes `existing` via the helper above, calls the existing `planImport(doc, existing, "overwrite")`, keeps only `kind === "overwrite"` actions, and for each reads the *current* file the same way the rest of the codebase already does (`readPromptFromCache` + `stripFrontmatter(await app.vault.cachedRead(file))`, both already exported from `storage/frontmatter.ts`), turns that into an `ExportedPrompt` via `toExportedPrompt`, and calls `diffImportEntry`. Reads only, no writes, so FR-17.1 ("before anything is written") holds structurally, not by convention. If a planned-overwrite target has vanished from disk by read time, that entry is silently dropped from the preview: `runImport` already falls back to creating in that exact situation, so dropping it keeps the preview consistent with what will actually happen instead of inventing a third code path for an already-handled edge case.

**UI:**

- New `src/ui/import-preview-modal.ts`: `ImportPreviewModal extends Modal`, constructed with `(app, diffs: ImportDiff[], onConfirm: () => void)` — the narrow-deps pattern already used by `VariableModal`/`ConfirmModal`, nothing plugin-shaped. Renders one block per `ImportDiff` (path heading, "identical" label or the field-change list formatted here — `tags.join(", ")`, `quality ?? "none"`, etc. — plus the body line), and a `Setting` row with Confirm (`setCta()`, closes then calls `onConfirm`) and Cancel (closes, no callback). This is a new file, not a generalization of `ConfirmModal` (whose single-`message: string` constructor does not fit a structured list) and not a change to `ConfirmModal`'s existing delete-confirmation usage in `library-view.ts`.
- `src/ui/import-modal.ts`: `submit()` validates exactly as today; when `policy === "overwrite"`, it awaits `buildOverwritePreview` and, if it returns at least one entry, opens `ImportPreviewModal` on top of the still-open `ImportModal` and returns without importing. Every other path (skip, duplicate, or overwrite with zero conflicts) falls through to a new private `runAndReport(folder, doc, policy)` that is exactly today's tail end (`runImport` → `close()` → `Notice` → `console.warn` on failures) extracted verbatim, called from both the direct path and from `ImportPreviewModal`'s `onConfirm`. `ImportModal` is deliberately left open underneath the preview (not closed first): on cancel the user lands back on their still-filled-in import form with nothing written and no summary shown, which satisfies the acceptance criterion without adding a "did we already close" branch to `runAndReport`.
- `styles.css`: additive rules for the preview list following the existing `promptbox-<component>__<part>` naming already used for `promptbox-import__errors` etc. (e.g. `promptbox-diff-preview__item`, `promptbox-diff-preview__field`, `promptbox-diff-preview__body`, `promptbox-diff-preview__identical`), Obsidian CSS variables only, no new dependency.

No change to `schema_version`, `ExportedPrompt`, `ExportDoc`, `planImport`'s signature, `runImport`'s signature, or `ImportSummary`. Zero network calls anywhere in the new code (ADR-0001/0002 binding, `docs/spec.md` NFR scope).

## Alternatives considered

1. **Diff against the raw `Prompt` domain type instead of reusing `ExportedPrompt`.** Would let the diff engine skip the `toExportedPrompt` extraction. Rejected: forces the comparator to bridge two different field-naming conventions (`useCase` vs `use_case`) and to explicitly blacklist `Prompt`-only fields (`custom`, `warnings`, and any future frontmatter field such as the `favorites` boolean landing on a sibling branch) so they never leak into "metadata field changes." Reusing the already-curated `ExportedPrompt` shape means any field the transfer schema does not know about is excluded for free, with no per-field allowlist to keep in sync as `Prompt` grows.
2. **Push the preview gate inside `runImport` itself, as a mid-run callback invoked right before the batch of overwrite actions.** Rejected: `planImport`'s output interleaves create/skip/overwrite/duplicate actions in document order rather than grouping by kind, so a create earlier in the array could already run before such a callback fires — a real risk of contradicting "before anything is written" if the loop ordering is not handled with extreme care. It also widens `runImport`'s signature for the skip/duplicate policies that will never use the callback. Keeping `runImport` untouched and putting the gate one layer up, in the modal, is a strictly smaller and safer change for an S-effort feature, and trivially guarantees FR-17.4 ("other policies untouched") by construction rather than by a conditional deep inside the write loop.
3. **Read the "existing" side of the diff from the in-memory `PromptIndex` instead of re-reading the file from disk.** Would avoid one redundant `cachedRead` per conflict. Rejected: introduces a new cross-module dependency from `transfer-io.ts` (currently obsidian + `domain/*` + `storage/frontmatter.ts` + `storage/slug.ts` only) onto `storage/indexer.ts`, and risks a staleness mismatch if the index has not caught up with the latest vault event by the moment of import — whereas reading straight from `app.vault`/`app.metadataCache`, the same way `runImport` and `readPromptFromCache` already do, guarantees the preview reflects exactly what the write step is about to see.
4. **Close `ImportModal` before opening `ImportPreviewModal`, so exactly one modal is ever visible.** Rejected: on cancel this would strand the user with no import dialog at all and no way back to their already-filled-in source/paste/policy fields without starting over, for no offsetting benefit; it would also force `runAndReport` to guard against being invoked with `ImportModal` already closed (harmless in Obsidian's `Modal.close()` almost certainly, but unverified and untested in this codebase) for the confirm path, while still needing the un-guarded call for the direct path. Leaving `ImportModal` open underneath the preview removes both problems: modal stacking is a supported native pattern, and every call to `runAndReport` finds `ImportModal` in the same still-open state.

## Consequences

**Positive:** the overwrite policy's destructive branch is de-risked exactly as `docs/competitive-analysis.md` §6 N5 intends, with zero change to the skip/duplicate policies or the end-of-run summary (FR-17.4, FR-17.5 hold by construction, not by extra runtime checks); `toExportedPrompt` becomes the one place that maps a `Prompt` + body into the transfer shape, ready to be reused by the parked N7 "curated packs" work and by the Phase 2 community-update diff (CFR-5.2) without redesign; the new domain functions are fully vitest-covered and the storage/UI layer changes are additive and small (no existing function signature changes, so blast radius stays inside `transfer-io.ts` internals, one new UI file, and `ImportModal.submit()`).

**Negative and accepted:** a narrow TOCTOU gap exists between `buildOverwritePreview`'s read and `runImport`'s later write — if the vault changes in between (external edit, sync, another pane), the preview can go slightly stale; not engineered around, consistent with the spec's explicit exclusion of three-way merge and per-item accept/reject. The vanished-file edge case (a previewed overwrite target deleted before `runImport` runs) means the preview and the actual run could, in a narrow race, disagree by one entry; accepted for the same reason. Including `created`/`updated` in the diffable field set is a reading of FR-17.2 ("which metadata fields change") beyond the single worked example in the acceptance criteria (which only shows `quality`); the justification is that it mirrors exactly what `applyEntry` already silently overwrites, but it is flagged here as a judgment call worth a human sanity check before merge, not a literal restatement of the spec text.

**Neutral:** `ImportPreviewModal` is a new, separate `Modal` rather than a generalization of `ConfirmModal`; this adds one file instead of widening a shared one, trading a small amount of duplication (title/Setting/button boilerplate already present in every modal in `src/ui/`) for zero risk to `ConfirmModal`'s existing delete-confirmation call site. Stacking a second native `Modal` on top of `ImportModal` while it stays open is visually a double backdrop on both desktop and mobile; this is standard Obsidian behavior, not a new pattern this codebase invents, but has no existing precedent in this repo's own modals to compare against and should get a manual look on both platforms.
